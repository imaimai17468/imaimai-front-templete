# Architecture Decision Records (ADR)

Long-term records of the rationale behind non-obvious design decisions. Plans and commit messages record *what changed*; an ADR records **why that option was chosen**, so a future reader can re-evaluate the decision when circumstances change.

## When to write an ADR

Write one when **all** of the following hold:

- The decision is hard to unwind without coordinated work
- The right answer was not obvious and credible alternatives existed
- The reasoning will be forgotten in six months

If the decision can be re-derived from the code or commit messages alone, no ADR is needed.

## Format

Use MADR-lite. Create each ADR at `docs/adr/NNNN-kebab-title.md` with this skeleton:

```markdown
# NNNN. Title

- Status: proposed | accepted | accepted (amended by NNNN) | superseded by NNNN | deprecated
- Date: YYYY-MM-DD

## Context

What problem is being solved? What constraints and forces are at play?

## Decision

What was decided. One paragraph, written declaratively.

## Alternatives considered

- **Option A**: why it was rejected
- **Option B**: why it was rejected

## Consequences

What becomes easier? What becomes harder? What follow-up is needed?
```

Keep each ADR to roughly 80 lines; split it if it grows longer.

## Numbering

Strictly sequential, four zero-padded digits. Never renumber. When an ADR is replaced, mark the old ADR `superseded by NNNN` and create a new one. When only part of a decision is revised by a later ADR, mark it `accepted (amended by NNNN)` and add a note at the top pointing to the amendment. In both cases, never rewrite the old ADR's Decision.
